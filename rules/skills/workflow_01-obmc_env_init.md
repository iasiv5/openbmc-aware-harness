# OpenBMC 开发环境初始化与构建

## 元数据

- **类型**: Workflow
- **适用场景**: 初始化 OpenBMC 开发环境（bare mirror 填充、构建配置生成）和编译固件镜像
- **创建日期**: 2026-05-31
- **最后更新**: 2026-06-08

---

## 目标

`ob init` 完成后，目标 machine 的 BitBake 构建环境处于就绪状态：所有 git 依赖的 bare mirror 已预填充到 `DL_DIR/git2/`，BitBake `do_fetch` 阶段可从本地 mirror 获取源码而不依赖远程网络。构建配置（缓存目录、连通性探针、WSL 并行度）已写入 `.inc` 文件并由 `local.conf` include。

`ob build` 从已初始化的 machine 中交互选择一个，执行 `bitbake obmc-phosphor-image` 编译固件镜像。

## 边界

**ob init 做**：
- Clone OpenBMC 主仓库（首次）
- 通过 `bitbake -g` + Tinfoil API 生成依赖图（deps.json）
- 预填充 bare mirror 到 `DL_DIR/git2/`（BitBake 标准 mirror 路径）
- 生成 lockfile 记录版本快照
- 生成 `externalsrc-<machine>.inc`（缓存目录、连通性探针、externalsrc 继承）
- 支持 `--dry-run`、`--skip-deps`、`--verbose`、`--url` 选项

**ob init 不做**：
- 不创建 working tree（`workspace/src/<machine>/` 不被 init 产生）
- 不注入 `EXTERNALSRC_pn-*` 映射（`.inc` 中 `INHERIT += "externalsrc"` 但无实际条目，为未来 `ob dev` 预留）
- 不做编译（`ob build` 是独立命令）
- 不处理 OpenBMC layer 的自定义 patch
- 不在 Windows 上运行（需要 Linux + bitbake 环境）

**ob build 做**：
- 发现所有已初始化的 machine（通过 `.init-done` marker）
- 交互选择并编译 `obmc-phosphor-image`
- 报告编译结果和镜像路径/大小

**ob build 不做**：
- 不自动选择 machine（必须交互确认）
- 不处理编译失败后的 recipe 级别清理

## 验收标准

`ob init` 完成后，以下条件应全部满足：

1. `workspace/openbmc/.git` 存在
2. `workspace/openbmc/build/<machine>/conf/local.conf` 中 `MACHINE = "<machine>"` 已设置
3. `workspace/openbmc/build/<machine>/conf/local.conf` 末尾包含 `include externalsrc-<machine>.inc`
4. `workspace/openbmc/build/<machine>/conf/externalsrc-<machine>.inc` 存在，包含 `DL_DIR`、`SSTATE_DIR` 和 `INHERIT += "externalsrc"`
5. `workspace/openbmc/build/<machine>/deps.json` 存在，是合法 JSON 且为 recipe 数组
6. `workspace/configs/<machine>.lock` 存在，是合法 JSON 且包含 `machine`、`openbmc_commit`、`sub_repos` 字段
7. `workspace/configs/<machine>.init-done` 存在（全部 8 步完成的原子 marker）
8. `workspace/downloads/git2/` 下存在若干 bare mirror 目录（数量取决于 deps.json 中 git 依赖的数量）

## 可用资源

**工具**：
- 根目录 `ob` — 主入口 bash 脚本，支持三种运行模式：
  - `./ob`（无参数）→ 交互菜单（init/build/status/clear/quit）
  - `./ob init [<machine>]` → CLI 模式初始化
  - `./ob build` → CLI 模式编译
- `tools/parse_bitbake_deps.py` — Python 脚本，用 BitBake Tinfoil API 批量查询 SRC_URI/SRCREV，从 `pn-buildlist` 读取 recipe 列表

**命令行选项**：

```bash
# 交互菜单（无参数）
./ob

# init 子命令
./ob init                          # 列出可用 machine 并交互选择
./ob init <machine>                # 基本用法
./ob init <machine> -d             # 预览模式（dry-run），不实际执行
./ob init <machine> -s             # 跳过依赖解析（skip-deps，应急选项）
./ob init <machine> -v             # 详细输出
./ob init <machine> -u <url>       # 指定 OpenBMC 主仓库 URL

# build 子命令
./ob build                         # 交互选择已初始化的 machine 并编译

# status 子命令
./ob status                        # 显示主仓库绑定状态
```

**锁文件 schema**（`workspace/configs/<machine>.lock`）：

```json
{
  "machine": "<machine>",
  "generated_at": "ISO-8601 timestamp",
  "openbmc_commit": "40-char SHA",
  "target_image": "obmc-phosphor-image",
  "sub_repos": [
    {
      "name": "phosphor-dbus-interfaces",
      "src_uri": "git://github.com/openbmc/phosphor-dbus-interfaces",
      "srcrev": "40-char SHA or AUTOREV",
      "local_path": "workspace/src/<machine>/phosphor-dbus-interfaces",
      "recipe": "phosphor-dbus-interfaces"
    }
  ]
}
```

> **注意**：`local_path` 是为未来 `ob dev` 预留的占位符，`ob init` 不会在该路径下创建文件。

**externalsrc inc 文件格式**（`conf/externalsrc-<machine>.inc`）：

```makefile
# Auto-generated by ob init <machine> at <timestamp>

# Enable externalsrc class (no-op when no EXTERNALSRC_pn-* is set).
# Use 'ob dev <recipe>' to add externalsrc entries for specific recipes.
INHERIT += "externalsrc"

# Disable OE connectivity probes
CONNECTIVITY_CHECK_URIS = ""

# Build cache directories
DL_DIR ??= "/absolute/path/to/workspace/downloads"
SSTATE_DIR ??= "/absolute/path/to/workspace/sstate-cache"

# [WSL only] WSL parallelism auto-tuning
BB_NUMBER_THREADS ?= "<N>"
PARALLEL_MAKE ?= "-j <N>"
```

## 约束

- **操作系统**：必须 Linux（bitbake 依赖）
- **前置工具**：git, python3
- **网络**：首次运行需访问 github.com 或企业 GitLab；`--skip-deps` 仅复用已有 `deps.json` 跳过依赖解析，bare mirror 填充阶段仍需网络
- **连通性探针**：`.inc` 文件设置 `CONNECTIVITY_CHECK_URIS = ""`，避免受限网络下被 OE sanity probe 误拦截
- **GitLab SSH 自动配置**：`ob init` 检测 GitLab IP（优先级：meta-\*/git-mirror-url.sh → git remote origin URL），自动配置 `git config --global url.git@<ip>:.insteadOf https://<ip>/`，解决 recipe 用 HTTPS 但服务器仅开放 SSH 的场景
- **WSL 并行度**：WSL 环境下自动计算 `BB_NUMBER_THREADS`/`PARALLEL_MAKE`（`(MemTotal+SwapTotal)/4`，cap at nproc），防止 OOM
- **磁盘空间**：≥ 30GB（主仓库 + bare mirror + 编译输出）
- **幂等性**：重复运行 `ob init <machine>` 是安全的——已存在的 bare mirror 会被跳过，配置文件会备份后覆盖
- **不要手动编辑** `.inc` 文件，需要变更时重新运行 `ob init`

## 已知陷阱

- **BitBake 赋值操作符优先级**：`.inc` 文件中覆盖 OE-core 默认值时，必须选对操作符。优先级（从弱到强）：`??=` < `?=` < `=`。OE-core 在 `bitbake.conf` 用 `?=` 设置 `BB_NUMBER_THREADS`/`PARALLEL_MAKE`，如果 `.inc` 用 `??=` 写自定义值，会被 OE-core 的 `?=` 覆盖（`??=` 是最弱赋值，只在没有其他赋值时生效）。正确做法：用 `?=`（同级后写者胜）或 `=`（强覆盖）。`??=` 只适用于"确认没有任何人设过"的场景。
- **bare mirror 失败不阻塞编译**：个别 mirror clone 失败时 `ob init` 只记录 warning，不退出。BitBake `do_fetch` 阶段会从远程 URL 直接拉取。如果远程也不可达，编译时才会报错。

## 输出文件结构

```raw
workspace/
├── openbmc/                                    # 主仓库
│   └── build/<machine>/                        # bitbake 构建目录
│       ├── conf/
│       │   ├── local.conf                      # MACHINE + externalsrc include
│       │   └── externalsrc-<machine>.inc       # 缓存目录 + externalsrc 继承
│       └── deps.json                           # 依赖解析结果（recipe → SRC_URI/SRCREV）
├── downloads/
│   └── git2/                                   # bare mirror 缓存（BitBake 标准 gitsrcname 命名）
│       ├── github.com.openbmc.phosphor-dbus-interfaces/
│       ├── github.com.openbmc.linux/
│       └── ...
├── sstate-cache/                               # 共享 sstate 缓存
└── configs/
    ├── openbmc-source.lock                     # 主仓库来源绑定（单 harness 单源）
    ├── <machine>.lock                          # 版本锁定快照
    └── <machine>.init-done                     # init 完成原子 marker
```

## 故障排除（Troubleshooting）

### ob init 阶段

#### 1. bare mirror clone 失败

**症状**：Step 5 报告 `[FAIL] <repo> (bare mirror clone failed)`
**影响**：不阻塞 init 流程，BitBake 编译时从远程拉取
**处理**：

```bash
# 查看失败详情
cat workspace/openbmc/build/<machine>/clone-errors.log

# 重试：删除失败的 mirror 目录后重跑
rm -rf workspace/downloads/git2/<gitsrcname-of-failed-repo>
./ob init <machine> -s    # -s 跳过依赖解析，直接重试 mirror 填充
```

#### 2. URL 不可达（特定域名）

**症状**：`could not determine hash algorithm` 或连接超时（如 infradead.org）
**处理**：`ob` 内置 URL 重写表（当前仅 `infradead.org → github.com/sigma-star`）。新域名需在 `clone_sub_repos()` 的 `_url_rewrites` 数组中添加条目。

#### 3. BitBake 变量未展开

**症状**：`[WARN] Unresolved BitBake variable in clone_url for <repo>: ${GITLAB_IP}/...`
**原因**：`deps.json` 中某些 recipe 的 `clone_url` 含 `${GITLAB_IP}` 等 BitBake 变量引用，但环境变量、git-mirror-url.sh 和 local.conf 中都找不到该变量的值
**处理**：在 `local.conf` 中设置该变量后重跑 `ob init`

#### 4. 主仓库来源冲突

**症状**：`A harness may only bind to one OpenBMC main repository source`
**原因**：`openbmc-source.lock` 绑定的源与请求的 URL 不一致
**处理**：复制 ob-harness 到新目录，在新目录中 `ob init` 使用不同 URL

### ob build 阶段

#### 5. 没有可构建的 machine

**症状**：`No machines are ready to build`
**原因**：没有 `.init-done` marker
**处理**：先运行 `./ob init <machine>`

#### 6. 编译失败

**症状**：`bitbake exited with code <N>`
**处理**：

```bash
# 方法 1：重试
./ob build    # 选择同一 machine

# 方法 2：清理失败 recipe 的 sstate 后重试
cd workspace/openbmc && source setup <machine>
bitbake -c cleansstate <failed-recipe>

# 方法 3：查看完整编译日志
ls workspace/openbmc/build/<machine>/tmp/log/cooker/<machine>/
```

## 已知限制

| 限制 | 原因 | Workaround |
|------|------|-----------|
| `mtd-utils` 从 infradead.org 不可克隆 | 部分构建服务器无法访问 infradead.org | 内置 URL 重写至 GitHub mirror |
| `json_test_data` commit 丢失 | 上游 force push 重写历史 | 使用最新 master HEAD |
| 个别 bare mirror 失败不阻塞 init | 设计决策：允许部分失败，BitBake 按需从远程拉取 | 重跑 init 或手动补 mirror |
| `--skip-deps` 需要已有 deps.json | 跳过 bitbake -g + Tinfoil，直接复用缓存 | 先完整跑一次 init |
| `ob build` 需要交互终端 | 必须交互选择 machine + Y/N 确认 | 无 TTY 环境下手动 `source setup && bitbake` |
