# OpenBMC 主仓单一来源锁定（single-source lock）—— 设计文档

> 日期：2026-06-02
> 状态：已批准，已实现
> 关联：`tools/ob`（`ob init <machine>`）、`docs/specs/2026-05-31-obmc-env-init-design.md`

## 背景与目标

`ob init <machine>` 把 OpenBMC 主仓固定克隆到单一路径 `workspace/openbmc/`（`OPENBMC_DIR` 是常量，见 `tools/ob`）。主仓来源可来自三处：`--openbmc-url`、环境变量 `OB_OPENBMC_URL`、交互式选择「社区 / 自定义 URL」。

现有 `clone_openbmc()` 只检测 `workspace/openbmc/.git` 是否存在：存在即**静默跳过克隆**，并提示用 `git pull` 更新。它**不记录、也不校验**这份主仓当初来自哪个来源。

**根因**：主仓没有「来源身份指纹」。因此当用户已克隆社区版后又想拉内网版（或反之），脚本会静默复用旧树，来源冲突无法被察觉。

**目标**：把「一个 harness 只绑定一个 OpenBMC 主仓来源」固化为硬契约。`ob init` 在复用现有主仓前，若本次显式给出的来源与已绑定来源不一致，则**在 clone 前停下**，报错并建议用户另开一份 harness copy 去下载不同来源（如内网版）。

**非目标**：
- 不支持在同一 harness 内并存多棵 openbmc 树
- 不提供同一 harness 内的「强制换源」开关（换源必须另开 harness copy）
- 不做更深层的篡改审计或自动修复（只检测 `openbmc-source.lock` 与 `git origin` 是否漂移）
- 不改变现有目录结构、构建流程、子仓库拉取逻辑

## 范围与非范围

**范围**：`tools/ob` 的 `clone_openbmc()` 来源比对与篡改检测逻辑、来源锁文件的生成与读取、URL 规范化函数、新增 `ob status` 只读子命令、相关错误提示与帮助文档。

**非范围**：多 machine 复用、子仓库来源校验、CI 流程、Windows wrapper。

## 方案比较与推荐方案

判定「来源是否一致」的粒度，已确认采用 **B：规范化 host + repo 路径**（忽略协议、`.git` 后缀、大小写）。

来源身份的存储与比对方式，三方案比较：

| 方案 | 核心思路 | 优点 | 缺点 |
|---|---|---|---|
| A：仅读 git origin | init 时跑 `git remote get-url origin` 比对，不落文件 | 零新增文件、无状态 | 依赖 git 元数据、用户改 origin 可绕过、harness 无声明式来源可读 |
| **B：来源锁文件（推荐）** | 首次克隆后写 `openbmc-source.lock`，init 先读它比对，无锁则回退读 origin 补写 | 声明式、可读、对 AI 友好、复用 `workspace/configs/` 约定 | 多一个文件和一次写盘、需处理锁与 origin 不一致的边角 |
| C：双重校验（推荐） | 在 B 基础上要求锁文件、git origin、本次请求三者一致 | 最严、能抓篡改 | 逻辑/分支变多 |

**推荐并已确认：方案 C（B + 篡改检测）**。以方案 B 的来源锁文件为声明式事实源，复用现有 `workspace/configs/` 约定、对 AI 友好；在此之上增加「lock 与现有树 git origin 一致性」的篡改检测，能抓出手动改 `origin` 或手动换树导致的来源漂移。代价是多一条校验分支，可接受。

## 已确认的设计决策

| # | 决策点 | 结论 |
|---|---|---|
| 1 | 目标场景 | B 场景：路径单一，但来源身份冲突需被检测并阻止 |
| 2 | 单主仓是否设为硬约束 | 是。「一个 harness 一个来源」是当前架构的隐含契约，予以固化 |
| 3 | 来源一致性判定粒度 | 规范化 host + repo path（去协议、去 `.git`、去末尾斜杠、去用户名@与端口、转小写） |
| 4 | 来源身份存储 | 来源锁文件 + 篡改检测（方案 C） |
| 5 | 换源逃生口 | 不提供。换源必须另开 harness copy |
| 6 | 锁文件路径 | `workspace/configs/openbmc-source.lock` |
| 7 | 锁文件格式 | 纯文本 `key=value`（不引入 YAML 依赖） |
| 8 | 来源标签 `source_label` | 保留该字段，纯记录、不参与比对 |
| 9 | 标签输入入口 | 交互选择自动赋值：社区→`community`，自定义 URL→`custom`；非交互的 `--openbmc-url`/`OB_OPENBMC_URL` 留空 |
| 10 | 本次来源比对触发条件 | 仅当本次**显式**给出来源（`--openbmc-url` 或 `OB_OPENBMC_URL`）时才做「本次来源 vs lock」比对；裸 `ob init <machine>` 不做此项 |
| 11 | 篡改检测触发条件 | 只要有 lock 且现有树有 `origin`，每次 init **都做**「lock vs git origin」校验，不一致即停（与本次是否显式给来源无关） |
| 12 | 只读状态命令 | 提供 `ob status`，展示当前绑定来源、标签、原始 URL 与 git origin 是否一致 |

## 锁文件设计

**路径**：`workspace/configs/openbmc-source.lock`

`workspace/` 整体 gitignore，因此锁文件天然不进版本库——它属于「绑定本地这棵树」的本地状态，符合语义。

**格式**（纯文本 `key=value`，自动生成，不应手改）：
```raw
# openbmc source lock — auto-generated by `ob init`, do not edit by hand
normalized_source=github.com/openbmc/openbmc
origin_url=https://github.com/openbmc/openbmc.git
source_label=
machine_first_init=romulus
created_at=2026-06-02T07:28:16Z
```

字段职责：

| 字段 | 是否参与比对 | 说明 |
|---|---|---|
| `normalized_source` | ✅ 唯一比对依据 | host + path，规范化后的来源指纹 |
| `origin_url` | ❌ | 原始 URL，供人读与排错 |
| `source_label` | ❌ | 可读分类（`community` / `custom` / 空） |
| `machine_first_init` | ❌ | 首次初始化的 machine，溯源用 |
| `created_at` | ❌ | 生成时间戳（UTC ISO8601），溯源用 |

## 规范化规则（判定粒度 B 的具体定义）

输入 URL → 输出 `host/path`：

1. 去掉协议头：`https://`、`http://`、`ssh://`、`git://`
2. 处理 `git@host:path` 形式 → `host/path`
3. 去掉用户名 `@`、端口号
4. 去掉末尾 `.git`、末尾 `/`
5. 全部转小写

示例（判为同源）：

- `git@github.com:openbmc/openbmc.git` → `github.com/openbmc/openbmc`
- `https://github.com/openbmc/openbmc` → `github.com/openbmc/openbmc`

## 控制流（`clone_openbmc()` 内）

```raw
ob init <machine> [--openbmc-url / OB_OPENBMC_URL / 交互]
        │
        ├─ workspace/openbmc/.git 不存在 ──► 正常 clone
        │        └─► clone 成功后：写 openbmc-source.lock
        │             (normalized_source、origin_url、source_label、machine、时间戳)
        │
        └─ workspace/openbmc/.git 已存在 ──► 进入「来源比对」
                 │
                 ├─ 读 lock 文件
                 │     ├─ 无 lock ──► 回退读 git origin，规范化后补写 lock
                 │     │             （origin 缺失则 warn 跳过补写），放行（跳过 clone）
                 │     └─ 有 lock ──► 取 lock.normalized_source
                 │
                 ├─ 篡改检测：读现有树 git origin，规范化
                 │     ├─ origin 缺失 ──► warn，跳过本项检测
                 │     ├─ 与 lock.normalized_source 一致 ──► 继续
                 │     └─ 不一致 ──► 🛑 停止（lock 与实际 origin 漂移，疑似手动改动）
                 │
                 ├─ 本次是否显式给了来源？(--openbmc-url / OB_OPENBMC_URL)
                 │     ├─ 否 ──► 放行（复用现有树，保持现状体验）
                 │     └─ 是 ──► 规范化本次来源，与 lock.normalized_source 比对
                 │             ├─ 一致   ──► 放行（跳过 clone，保持现有 git pull 提示）
                 │             └─ 不一致 ──► 🛑 停止，报错并建议另开 harness copy
```
三个关键判定点：

1. **篡改检测每次都做**（只要有 lock 且现有树有 origin）。它校验「声明的来源」与「实际 git origin」是否漂移，与本次是否显式给来源无关。
2. **本次来源比对只在显式给出来源时做**。裸 `ob init <machine>`（无 url、无环境变量）意图是「复用现有主仓」，不弹交互、不拦截，保持现有体验。
3. **任一不一致都在 clone 前就停**，绝不覆盖、绝不静默改 origin。

## 错误处理与边角情况

| 情况 | 行为 |
|---|---|
| 篡改检测失败（lock vs 现有树 git origin） | 🛑 clone 前停止，打印「lock 声明来源 / 实际 origin / 二者不一致」，提示疑似手动改动，退出非 0 |
| 有 `.git` 但无 lock（旧 harness 升级） | 回退读 git origin，规范化后补写 lock（`source_label` 留空），放行 |
| 有 `.git`、有 lock，本次未给显式来源 | 不比对，放行（复用现有树） |
| lock 存在但 `.git` 不存在（树被删、残留 lock） | 视为需重新绑定：忽略旧 lock，正常 clone 后用本次来源覆写 lock |
| `git remote get-url origin` 失败（无 origin） | 补写场景下跳过补写并打印 warn，仍放行（不阻塞） |
| `--dry-run` | 只打印将写入/比对的内容，不实际写 lock、不实际克隆 |

## 错误提示文案（不一致时）
```raw
[ERROR] OpenBMC main repository source conflict detected.
  Current harness source: github.com/openbmc/openbmc
  Requested source:       internal.example.com/bmc/openbmc
A harness may only bind to one OpenBMC main repository source.
To use a different source, copy openbmc-aware-harness and run ob init in the new copy.
```
篡改检测失败时：

```raw
[ERROR] OpenBMC main repository source does not match the lock file.
  Locked source:  github.com/openbmc/openbmc
  Actual origin:  internal.example.com/bmc/openbmc
Existing openbmc origin and openbmc-source.lock have drifted, likely due to a manual change.
Please verify the main repository source before continuing. If you need a different source, use a separate harness copy.
```
## `ob status` 子命令

新增只读子命令 `ob status`，展示当前 harness 绑定的 OpenBMC 主仓来源，便于人/AI 一眼确认绑定情况，无副作用、不修改任何文件。

行为：

- 读 `workspace/configs/openbmc-source.lock` 与现有树 `git origin`
- 展示：绑定来源（`normalized_source`）、标签（`source_label`）、原始 URL（`origin_url`）、首次初始化 machine、绑定时间、当前 git origin、以及二者是否一致
- 退出码：一致或信息齐全 → 0；篡改（lock 与 origin 不一致）→ 非 0，便于脚本判断
各状态输出：

| 状态 | 输出 | 退出码 |
|---|---|---|
| 无 lock、无 `.git` | 提示 `Not initialized` 与 `Run ob init <machine>` | 0 |
| 无 lock、有 `.git` | 提示 `Repository present without source lock`，并展示当前 origin 与 `Re-run ob init to bootstrap openbmc-source.lock` | 0 |
| 有 lock、无 `.git` | 展示 lock 字段，并提示 `Main repository not cloned or was deleted` | 0 |
| 有 lock、有 `.git`、origin 与 lock 不一致 | 展示双方来源，标注 `origin drift detected` 与 `Lock file and origin do not match` | 非 0 |
| 有 lock、有 `.git`、origin 与 lock 一致 | 展示绑定来源全部字段，标注 `origin matches` 与 `OK` | 0 |

输出示例（一致）：

```raw
OpenBMC main repository binding status
  Bound source:    github.com/openbmc/openbmc
  Origin URL:      https://github.com/openbmc/openbmc.git
  First init:      romulus  @ 2026-06-02T07:28:16Z
  Current origin:  github.com/openbmc/openbmc  (origin matches)
  Status:          OK
```

## 测试策略（手动 + dry-run，无单测框架）

- 首次 clone → 生成 lock，字段正确（含 `source_label`）
- 同源再 init（https vs ssh 写法）→ 判一致，放行
- 异源再 init → 拦截退出非 0，提示文案正确
- 旧 harness（有 `.git` 无 lock）→ 补写 lock，放行
- 旧 harness（有 `.git` 无 lock）真实执行一次 `./tools/ob init <machine> --skip-fetch` → 在 Step 2 补写 lock，后续继续现有 init 流程
- 删树留 lock → 重新 clone 覆写 lock
- 裸 `ob init <machine>`（无显式来源）→ 不做本次来源比对，但仍做篡改检测，放行
- 篡改检测：lock 与现有树 origin 不一致 → 拦截退出非 0，提示文案正确
- 篡改检测：现有树无 origin → warn 跳过，放行
- 全程兼容 `--dry-run`（只打印不写盘、不克隆）
- 规范化函数单点验证：协议差异、`git@` 形式、端口、末尾 `.git`/`/`、大小写
- `ob status`：五种状态（见上表）输出与退出码正确，且无副作用（不写 lock、不改 origin）

## 未决事项

- `source_label` 目前仅 `community` / `custom` / 空三态；若未来引入更细分类（如多个内网源）再扩展
- `ob status` 当前仅覆盖主仓来源绑定状态；未来若 `ob status` 扩展为展示 machine/build/子仓库整体状态，需统一输出格式
